# Multi-Feature Split Strategy

Detailed algorithm for `/commit --multi` — splitting a branch with multiple features into separate branches and PRs.

## Step 1: Gather Data

```bash
# All commits on this branch since main
git log main..HEAD --stat --format="%H %s"

# Full diff for understanding dependencies
git diff main..HEAD

# Current branch name
git branch --show-current
```

## Step 2: Identify Feature Boundaries

Use three signals, weighted by reliability:

### Signal 1: Module Boundaries (strongest)
Files in different top-level directories or domains are likely different features:
- `convex/billing/` vs `convex/workspaces/` → different features
- `components/landing/` vs `components/product/` → different features
- `app/(app)/settings/` vs `app/(app)/dashboard/` → different features

### Signal 2: File Co-occurrence in Commits
Files that always appear together in the same commits belong to the same feature. Build a co-occurrence matrix from `git log --stat`.

### Signal 3: Import/Type Dependencies
Trace import chains to find what depends on what:
- Shared infrastructure (schema.ts, validators.ts, utils.ts) may be needed by multiple features
- If a change is only imported by one feature's files, it belongs to that feature
- If a change is imported by multiple features, it's "shared" and goes into the first/base PR

## Step 3: Handle Shared Changes

Some changes are shared across features (schema updates, utility functions, type definitions). Strategy:
1. Identify shared files (imported by files in 2+ feature groups)
2. Create a "base" branch with shared changes first
3. Feature branches are created from main, with shared commits cherry-picked first
4. Alternatively: put shared changes in the first PR and note the dependency

## Step 4: Present Groupings

Show the user a table:
```
Feature 1: "Billing dashboard" (5 commits, 8 files)
  - convex/billing/mutations.ts
  - convex/billing/queries.ts
  - components/product/billing-panel.tsx
  - app/(app)/dashboard/billing/page.tsx
  ...

Feature 2: "Slack OAuth" (3 commits, 4 files)
  - convex/integrations/slack.ts
  - components/product/slack-connect.tsx
  ...

Shared (will be included in all PRs):
  - convex/schema.ts (new tables used by both features)
```

Wait for user confirmation. Allow them to move files between groups.

## Step 5: Create Branches and PRs

For each feature group:

```bash
# Create branch from main
git checkout -b feat/<feature-name> main

# Cherry-pick shared commits first (if any)
git cherry-pick <shared-commit-hashes>

# Cherry-pick or apply feature-specific commits
git cherry-pick <feature-commit-hashes>

# If cherry-pick fails due to conflicts (commits contain mixed changes):
# Fall back to creating a patch from the specific files and applying it
git diff main..original-branch -- <file1> <file2> | git apply
git add <file1> <file2>
git commit -m "type(scope): description"

# Push and create PR
git push -u origin feat/<feature-name>
gh pr create --title "type(scope): description" --body "$(cat <<'EOF'
## Summary
<auto-generated from commit messages and changed files>

## What changed
<file list with descriptions>

## Test plan
- [ ] Quality pipeline passes
- [ ] Feature-specific verification
EOF
)"
```

## Step 6: Report

After all branches and PRs are created:
```
Created 3 feature branches with PRs:

1. feat/billing-dashboard — PR #47
   5 commits, 8 files
   https://github.com/owner/repo/pull/47

2. feat/slack-oauth — PR #48
   3 commits, 4 files
   https://github.com/owner/repo/pull/48

3. feat/landing-redesign — PR #49
   2 commits, 3 files
   https://github.com/owner/repo/pull/49
```

Return to the original branch when done.

## Edge Cases

- **Commits that touch files in multiple features:** Split the commit conceptually — apply the relevant hunks to each feature branch separately
- **Merge conflicts during cherry-pick:** Fall back to file-level patching (git diff + apply for specific files)
- **Linear commit history required:** If commits are tangled, create fresh commits on each feature branch from the diff rather than cherry-picking
- **Base branch isn't main:** Detect with `git merge-base HEAD main` and use the actual diverge point
